数据级去重

在本章的前几节中我们介绍了如何通过一些特殊的手段建立高匿且能处理Javascript的爬虫，从防卫森严的花瓣网中抓取数据。回顾上一节蜘蛛对数据的提取逻辑，每个爬取请求返回时会取出一批的Item而不再像前几章书那样一个响应对象内只存有一个独立的Item。这种情况会带来一个新的问题：数据级去重。

数据级去重与URL去重完全不一样，URL是负责对发出请求之前对重复请求进行过滤降低爬虫所消耗的资源。但是在即使URL不重复的情况下，也会出现爬取数据重复。花瓣网就是这样的一个例子，页面加载的数据是随机生成的，那就有可能在加载完一批数据以后在后面的页面会重新出现已经加载过的数据。在爬虫中如果不进行处理就可能产生一大批重复的数据项。所以说我们在提取数据并进行存储之前就需要对数据项进行一次筛选去重。

在本章第一节的分析中我们得知，每个图片的容器元素div都带有一个data-id值，这是什么鬼？反过来推敲一下就很容易明白这一定是开发者想标识每个对象的ID，也就是它们在数据库中的ID值来的，这个值一定不会重复！所以我们可以在Item中增加这个字段，并且在蜘蛛中收集这个data-id值到Item实例中。

然后，我们将在第4章学到的Redis去重过滤器技术进行一些演化就能知道如何来做数据去重过滤器了。去重过滤器是在拦截URL这一级别的，但是数据过滤器则是在请求返回了响应对象对数据项(Item)进行处理这个时点的，所以两者一定是采用不同的方式来实现的。对数据进行处理当然是选择管道(Pipline)了，这一知识点我们在第4章学习大规模存储时开发的MongoDB存储管道就学习过了。所以我们可以将这两个技术点进行结合就可以得知，我们需要做数据去重的话就需要开发一个数据过滤管道，如果ID重复则不对Item进行下一步的处理。

# -*- coding: utf-8 -*-

import logging
from redis import Redis
from scrapy.exceptions import DropItem


class RedisDupePipeline(object):

    def __init__(self, host='localhost', key='IDFingerprints', port=6379, db=0):
        self.redis = Redis(host=host, port=port, db=db)
        self.key = key
        self.logger = logging.getLogger(__name__)

    @classmethod
    def from_crawler(cls, crawler):
        host = crawler.settings.get('REDIS_HOST', 'localhost')
        redis_port = crawler.settings.getint('REDIS_PORT')
        redis_db = crawler.settings.get('REDIS_DUP_DB')
        key = crawler.settings.get('REDIS_DUP_KEY')
        return cls(host, key, redis_port, redis_db)

    def process_item(self, item, spider):
      pass

先在pipelines.py文件中加入以上的RedisDupePipeline管道代码，这一步份的代码我们在之前的章节已学过，就是初始化一个Redis的实例，在此就不多作赘述了看出不懂的请回到第4章重温一下去重处理的内容。

接下来是对process_item这个方法的实现，原则很简单：如果在Redis存有的ID值的数据项就抛弃（引发一个DropItem异常)，不重复的则向下传递：

def process_item(self, item, spider):
  id = item['id']
  if not self.redis.sismember(self.key, id):
    self.redis.sadd(self.key, id)
    return item
  else:
  	raise DropItem(u'过滤重复项:%s' % id)

这样我们就保证了数据的唯一性，避免存储与处理重复性的数据，为以后避免下载重复图片内容做好充分备准。

如何爬取图片

对于多媒体文件（视频）和图片的抓取与下载我们并不需要自己来实现，Scrapy就搭载了项重要而且实用的功能。

ImagePipeline 的使用

Scrapy提供了一个专用图片管道ImagesPipeline来下载属于某个特定数据项目中的图片。以下是ImagesPipeline类的一些使用特性与能力:

  1. 将所有下载的图片转换成通用的格式（JPG）和模式（RGB）。
  2. 避免重新下载最近已经下载过的图片。
  3. 自动生成缩略图并保存成文件。
  4. 检测图像的宽/高，过滤掉不满足限制的图片。
  5. 这个管道也会为当前安排好要下载的图片保留一个内部队列，并将到达的包含相同图片的项目连接到这个队列中。这样可以避免多次下载几个项目共享的同一张图片。

ImagesPipeline通过Pillow来生成缩略图，并将图片归一化为JPEG/RGB格式。因此，为了使用图片管道，需要安装这个库。

使用图片管道

Scrapy内置的ImagesPipeline使用起来非常的简单，具体有两种做法：第一种是通过配置直接使用ImagesPipeline；第二种是继承ImagesPipeline改写存储逻辑。

先来看看ImagesPipeline的典型工作流程，了解一下它是如何运作的：

  1. 在一个蜘蛛里抓取一个Item，并把其中图片的URL存入img_urls属性中，这个属性是提供给图片管道进行读取的。
  2. 当Item进入ImagesPipeline时，img_urls数组内的URLs就会被Scrapy的调度器和下载器安排下载，当优先级更高时，会在其他页面被抓取前处理。项目会在这个特定的管道阶段保持“locker”的状态，直到完成图片的下载（或者由于某些原因未完成下载）。
  3. 当图片下载完，Scrapy将会得到一套本地的存储地址，此时就会自动存入到Item项的img_files 属性内，这也是一个数组对象，其中包含了一个字典列表，附带下载图片的信息，比如下载路径、源抓取地址（从img_urls属性获得）和图片的校验码。img_files列表中的图片顺序将和源img_urls组保持一致。如果某个图片下载失败，则记录下错误信息，图片也不会出现在img_files组中。

根据以上的原理我们就有必要先改造一下花瓣网的数据项结构与蜘蛛存储数据时的代码了：

# -*- coding: utf-8 -*-

from scrapy import Item, Field


class HuabanItem(Item):
    id = Field()
    img_urls = Field()
    img_files = Field()
    link = Field()
    desc = Field()

修改蜘蛛的分析赋值代码, 要将数据存到一个地址数组中而不是一个字符串中：

# huaban/spiders/bee.py

def parse(self, response):
  item = HuabanItem()
  pinEles = response.css("#waterfall div.pin")
  for pinEle in pinEles:
     # 此处略去，与原来代码一至，仅改动原img_url的部份
     item['img_urls'] = ['https:%s' % pinEle.css('a.img>img::attr(src)').get()]

  yield item

然后在配置文件settings.py中添加ITEM_PIPELINES配置项并指定为ImagesPipeline，具体如下所示。

# settings.py
ITEM_PIPELINES = {'scrapy.pipelines.images.ImagesPipeline': 1}

最后要将IMAGES_STORE配置项指定为一个有效的文件夹，用来存储下载的图片，否则管道将保持禁用状态。

例如:

# settings.py
# 指定图片的存储目录
IMAGES_STORE = '/Users/Ray/projects/huaban/images/'
# 在Item中存储目标图片下载地址的字段名
IMAGES_URLS_FIELD = 'img_urls'
# 当下载完后将下载结果对象写入到Item对象的字段名
IMAGES_RESULT_FIELD = 'img_files'
# 加载图片存储管道
ITEM_PIPELINES = {
  'huaban.pipelines.RedisDupePipeline': 1
  'scrapy.contrib.pipeline.images.ImagesPipeline': 2
}

完成以上配置后，在配置的时候一定要注意将IMAGES_URLS_FIELD 和 IMAGES_RESULT_FIELD与你的Item对象中定义的字段对上否则ItemPipeline就会工作异常。

此时图片下载的管道就已经配置成功了并且能正常地爬取与下载图了。接下来是对图片管道的其它的一些能力与配置的介绍，作为扩充阅读：

图片存储

图片会使用它们URL的SHA1 hash作为文件名。比如，对下面的图片URL：

http://www.example.com/image.jpg

它的SHA1 hash值为：

3afec3b4765f8f0a07b78f98c07b83f013567a0a

当文件下载完成后，会被保存成为下面的文件:

<IMAGES_STORE>/full/3afec3b4765f8f0a07b78f98c07b83f013567a0a.jpg

其中：

  • <IMAGES_STORE>是定义在IMAGES_STORE设置中的文件夹路径。
  • full是用来区分图片和缩略图（如果使用）的一个子文件夹。

图片的有效期

图像管道用来避免下载最近已经下载的图片。使用IMAGES_EXPIRES配置项调整失效期限，可以用天数来指定，具体如下所示。

# 90天的图片失效期限
IMAGES_EXPIRES = 90

当图片失效时，Scrapy就会重新下载该图片，反之则会跳过下载过程。

缩略图生成

图片管道可以自动创建下载图片的缩略图。为了使用这个特性，需要设置IMAGES_THUMBS配置项字典，其关键字为缩略图名字，值为它们的大小尺寸。

比如:

IMAGES_THUMBS = {
    'small': (50, 50),
    'big': (270, 270),
}

当使用这个特性时，图片管道将使用下面的格式来创建各个特定尺寸的缩略图：

<IMAGES_STORE>/thumbs/<size_name>/<image_id>.jpg

其中：

  • <size_name>是IMAGES_THUMBS字典的关键字（small、big等）。
  • <image_id>是图像URL的SHA1 hash.

例如，使用small和big缩略图名字的图片文件：

<IMAGES_STORE>/full/63bbfea82b8880ed33cdb762aa11fab722a90a24.jpg
<IMAGES_STORE>/thumbs/small/63bbfea82b8880ed33cdb762aa11fab722a90a24.jpg
<IMAGES_STORE>/thumbs/big/63bbfea82b8880ed33cdb762aa11fab722a90a24.jpg

第一个是从网站下载的完整图片。

过滤小图片

在IMAGES_MIN_HEIGHT和IMAGES_MIN_WIDTH设置中指定最小允许的尺寸就可以过滤那些过小的图片。

比如:

IMAGES_MIN_HEIGHT = 110
IMAGES_MIN_WIDTH = 110

注意：这些尺寸一点也不影响缩略图的生成。

默认情况下ImagesPipeline是没有尺寸限制的，因此所有图片都将被处理。

自定义图片管道

要自定图片管道最简单的方法就是继承自ImagesPipeline类，以下是它的类定义：

class scrapy.pipelines.images.ImagesPipeline

在上面的工作流程中可以看到，管道会得到图片的URL并从项目中下载。需要重写get_media_requests()方法，并对各个图片URL返回一个Request：

def get_media_requests(self, item, info):
    for image_url in item['image_urls']:
        yield scrapy.Request(image_url)

当然上述写法就是ImagesPipeline原有get_media_requests的代码，从中我们就可以了解为什么要在自定义的Item中定义image_urls。当我们遇到更复杂的情况时就可通过改写get_media_requests以产生图片下载请求。

如果get_media_requests()方法返回None，则意味着项目中没有图片可下载。

这些请求将被管道处理，当它们完成下载后，会以双元素元组列表形式传送到item_completed(results, items, info)方法的results参数。

以下为results参数中具有的键及其顺序：

  • success是一个布尔值，当图片成功下载时为True，因为某个原因下载失败为False。
  • image_info_or_error是一个包含下列关键字的字典（如果成功为则True，出问题时为Twisted Failure）。
  • url - 图片下载的URL。这是从get_media_requests()方法返回请求的URL。
  • path - 图片存储的路径（类似IMAGES_STORE）。
  • checksum - 图片内容的MD5 hash。

下面是results参数的一个典型值：

建立基于阿里云OSS的存储端

在做图片拉取项目时经常会遇到这样的困境：

  • 硬盘空间不够，由其是使用SSD最大的也不到1T，而图片一下就可能占掉好几百个G。
  • 要将收集的图片上传到其它的服务器，例如FTP的，AWS的又或者是阿里云的OSS。

Scrapy毕竟是老外开发的，对于国外的存储环境它是支持得比较充足，但对于国内FTP早已淘汰，AWS与GOOGLE又不能访问的情况来说这些内置的存储端则形同虚设。

相对于国外的这些存储空间而言，阿里云的OSS变得越来越流行，空间大不说速度也高还可以支持CDN最重要的是它便宜，大家都用得起。为了增加我们爬虫专栏的含金量与实用性，我们将示例扩展到云端为阿里云写一个存储端让我们的爬虫在收集到数据以后自动将图片上传到OSS中。

安装与使用 OSS sdk

阿里云上有OSS的官方SDK说明与众多的示例，但我们只需要其中的一小部份就可以驱动起一个存储后端了，用pip安装OSS的python SDK:

$ pip install oss2

让ImagesPipeline支持阿里云

ImagesPipeline的实现机制相当复杂，针对它scrapy还专门提供了一种文件存储机制用于将爬取后的图片存储到 本地、S3和Google Cloud。如果我们想将文件存储到阿里云比较理想的做法是扩展ImagesPipeline的存储机制。但遗憾的是，在图片文件存储方面官方完全没有公布任何资料。

我读完它的源码之后也找到了扩展的办法。ImagePipeline的存储接口设计得很差，又或者说根本没有考虑到要进行扩展。所以我们要增加一个新的存储机制只能继承ImagesPipeline后直接将基类的方法给覆盖，软的不行上硬的!

改写ImagesPipeline

首先我们从ImagesPipeline继承并创建一个新的管道类:

# -*- coding: utf-8 -*-

from scrapy.pipelines.images import ImagesPipeline as _ImagesPipeline
from scrapy.pipelines.files import FSFilesStore, S3FilesStore, GCSFilesStore
from .files import OSSFilesStore

class ImagesPipeline(_ImagesPipeline):
    STORE_SCHEMES = {
        '': FSFilesStore,
        'file': FSFilesStore,
        's3': S3FilesStore,
        'gs': GCSFilesStore,
        'oss':OSSFilesStore # 通过Scheme关联阿里云OSS
    }

    def __init__(self, store_uri, download_func=None, settings=None):
        super(ImagesPipeline, self).__init__(store_uri, settings=settings,
                                             download_func=download_func)

    @classmethod
    def from_settings(cls, settings):
        s3store = cls.STORE_SCHEMES['s3']
        s3store.AWS_ACCESS_KEY_ID = settings['AWS_ACCESS_KEY_ID']
        s3store.AWS_SECRET_ACCESS_KEY = settings['AWS_SECRET_ACCESS_KEY']
        s3store.POLICY = settings['IMAGES_STORE_S3_ACL']

        gcs_store = cls.STORE_SCHEMES['gs']
        gcs_store.GCS_PROJECT_ID = settings['GCS_PROJECT_ID']
        gcs_store.POLICY = settings['IMAGES_STORE_GCS_ACL'] or None
				# 阿里云OSS的文件存储的基本配置项
        ossStore = cls.STORE_SCHEMES['oss']
        ossStore.OSS_ACCESS_KEY = settings['OSS_ACCESS_KEY']
        ossStore.OSS_ACCESS_SECRET = settings['OSS_ACCESS_SECRET']

        store_uri = settings['IMAGES_STORE']
        return cls(store_uri, settings=settings)

这个类只有两个函数一个是构造函数另一个是读取配置用的from_settings函数，具体内容不用管理它照抄就好了，重点在于STORE_SCHEMES这个属性，在ImagesPipeline的父类(FilesPipeline)中是通过这个属性来获取文件存储类的。

FilesPipeline首先是通过判断IMAGE_STORE中指定的路径来判定我们是采用什么存储的，这个与我们之前讲存储端的时候Scrapy通过FEED_URI来判定存储端的类型的原理是一样的。

那我就利用它的这个原理，先添加一个OSSFilesStorage，然后在这个STORE_SCHEMES中添加存储协议与存储类的映射。在编写OSSFilesStorage以前我们就得在配置文件中加入以下的配置项，让我们实现的ImagesPipeline能实现这种关联：

IMAGE_STORE='oss://<Bucket名称>.<外网EndPoint>.aliyuncs.com/<子目录>'
OSS_ACCESS_KEY = 'OSS上访问公钥'
OSS_ACCESS_SECRET = 'OSS的访问私钥'

阿里云的目录访问是以下这个格式的：

https://<Bucket名称>.<外网EndPoint>.aliyuncs.com/<子目录>

我于是在IMAGE_STORE玩了个小把戏将https换成了oss这样原生的FilesPipeline就可以根据URI的scheme段来判定我们当前使用的是阿里云OSS作为文件存储。

由于访问阿里云OSS需要提供该存储的公钥与私钥（在阿里云的管理后台可以获得）。

我们可以在构造函数中通过分析URI的构成得到Bucket名称与EndPoint名称，然后构造Bucket的实例:

def __init__(self, uri):
  #oss: //< Bucket >.< 外网Endpoint > /< Object >
  u = urlparse(uri)
  self.uri = uri
  self.bucket_name = u.hostname.splite('.')[0]
  self.endpoint = '.'.join(u.hostname.splite('.')[1:])
  self.objectPath = u.path
  self.bucket = oss2.Bucket(oss2.Auth(self.OSS_ACCESS_KEY, self.OSS_ACCESS_SECRET),self.endpoint, self.bucket_name)

stat_file是个查询函数，只当文件上传成功后才执行，由于访问阿里云是个异步过程，这里就采用异步方式用回调来等待阿里云的调用返回：

def stat_file(self, path, info):

        def _onsuccess(meta):

        return threads.deferToThread(self.bucket.get_object_meta, path).addCallback(_onsuccess)

最后就是在presist_file函数中实现将文件分片上传到OSS

def persist_file(self, path, buf, info, meta=None, headers=None):
        # 首先可以用帮助函数设定分片大小，设我们期望的分片大小为128KB
        total_size = len(buf)
        part_size = oss2.determine_part_size(total_size, preferred_size=128 * 1024)

        # 初始化分片上传，得到Upload ID。接下来的接口都要用到这个Upload ID。
        key = os.path.join(self.objectPath, info)
        upload_id = self.bucket.init_multipart_upload(key).upload_id

        # 逐个上传分片
        # 其中oss2.SizedFileAdapter()把fileobj转换为一个新的文件对象，新的文件对象可读的长度等于size_to_upload
        parts = []
        part_number = 1
        offset = 0
        while offset < total_size:
            size_to_upload = min(part_size, total_size - offset)
            result = self.bucket.upload_part(key, upload_id, part_number,
                                             oss2.SizedFileAdapter(buf, size_to_upload))
            parts.append(oss2.models.PartInfo(part_number,
                                              result.etag,
                                              size=size_to_upload,
                                              part_crc=result.crc))

            offset += size_to_upload
            part_number += 1

            # 完成分片上传
            self.bucket.complete_multipart_upload(key, upload_id, parts)
